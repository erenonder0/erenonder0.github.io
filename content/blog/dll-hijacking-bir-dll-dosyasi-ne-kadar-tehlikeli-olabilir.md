<p>Almost every program we use daily calls dozens, maybe hundreds, of DLLs in the background. Much of the critical functionality of the operating system and of applications rests on these small files. But can these small, unremarkable files become a doorway to <strong>full system compromise in the wrong hands?</strong> In this post we will show just how common and invisible DLLs are, and then walk step by step through how that invisibility can turn into risk.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*rwcYLfiqdUfpSnFeTVOObg.png" /></figure><p><strong>DLL hijacking</strong> is a cyberattack technique that runs malicious code by manipulating the process a legitimate application uses to load a Dynamic Link Library (DLL).</p><h3><strong>What Is a DLL?</strong></h3><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/0*Mo9O7brDT7foXOx6.jpg" /></figure><p>A DLL (Dynamic Link Library) is a file containing reusable code and data that multiple programs can use at the same time to perform different functions, improving efficiency and modularity in software development. On Windows operating systems, most of the OS's functionality is provided by DLLs. This is why many DLL hijacking attacks stem from the operating system's and applications' heavy dependence on DLLs.</p><h4><strong>Why Are DLLs Used?</strong></h4><ul><li>Reuse (the same code is used by more than one application)</li><li>Ease of updating &amp; patching (when an update ships, only the relevant DLL is updated instead of the whole application)</li><li>Smaller files (the main .exe is kept minimal)</li></ul><h3><strong>How Does a DLL Work?</strong></h3><p>When an executable (.exe) is run and the program needs a DLL, the operating system searches for the required DLL file in certain directories. Once the file is found, it is loaded into memory and made available to the program.</p><h4><strong>DLL Search Order</strong></h4><p>When a DLL is needed, the DLL loader starts searching for it on the system.</p><p><strong>On Windows, the DLL search order is generally as follows:</strong></p><ol><li><strong>The directory the application was loaded from</strong></li><li><strong>The system directory (C:\Windows\System32)</strong></li><li><strong>The Windows directory (C:\Windows)</strong></li><li><strong>The current working directory</strong></li><li><strong>The directories listed in the Path environment variable</strong></li></ol><figure><img alt="" src="https://cdn-images-1.medium.com/max/600/1*INcO63doMuReaWOjyv6Mrg.png" /></figure><p>This ordering is critically important for DLL hijacking. During this search, we can inject our own malicious DLL into the location where the innocent DLL is expected. This is the very basis of the DLL hijacking attack.</p><h3><strong>What Is DLL Hijacking?</strong></h3><p>DLL hijacking is one of the serious threats in cybersecurity. Attackers run malicious code by manipulating the process a legitimate application uses to load its dynamic link libraries (DLLs). This method takes advantage of weak configurations such as misconfigured search paths or writable directories, and can be used for privilege escalation or to establish persistent access.</p><h3><strong>How Does DLL Hijacking Work?</strong></h3><p>The core idea is to abuse the search order the operating system uses to find and load DLLs. For example, if an application is looking for a library called "eren.dll", the attacker creates a DLL with the same name that carries a malicious payload inside it. If the original eren.dll sits in a directory that is checked later in the search order, the attacker places their malicious DLL in a directory the loader checks first. As a result, the operating system loads the attacker's DLL instead of the original file, and the malicious code is executed.</p><h3><strong>Common DLL Hijacking Techniques</strong></h3><ul><li><strong>Missing DLL Load</strong></li><li><strong>Search Order Hijacking</strong></li><li><strong>Phantom DLL Hijacking</strong></li><li><strong>DLL Sideloading</strong></li></ul><h4><strong>Missing DLL Load</strong></h4><p>This situation arises when a DLL an application needs is not present in the right place / is missing. The attacker can run malicious code by placing a fake copy of the missing DLL in the location being searched.</p><blockquote><strong>For example: the application eren.exe needs helper.dll and tries to load it, but if the installation is incomplete the loader will check the other search locations. If the attacker places helper.dll directly in the location where the file is searched for, the malicious code will run.</strong></blockquote><h4><strong>Search Order Hijacking</strong></h4><p>This is the most classic and quite common DLL hijacking method. The attacker exploits a program's DLL search order to make a malicious DLL with the same name load in place of the original DLL.</p><h4><strong>Phantom DLL Hijacking</strong></h4><p>This arises when an application searches for a DLL file that does not exist. If the attacker places their own malicious DLL in the location where the missing module is searched for, the application will run the file.</p><blockquote>For example: after an update, the application eren.exe no longer uses helper.dll directly. But if the application still tries to search for and load this DLL, the attacker can make the application load malicious code by placing a malicious DLL with the same name in the location where the DLL is searched for.</blockquote><h4><strong>DLL Sideloading</strong></h4><p>This technique copies a legitimate application, together with its folder, into a directory where read/write permissions are open, so that a malicious version of the DLL it runs from its current folder gets executed. For example, an application called eren.exe loads and uses helper.dll from its own directory when run. If the attacker places a helper.dll carrying a malicious payload — with the same name as the DLL eren.exe runs — into a different folder with write/read permissions such as tmp, and then runs the application from that tmp folder, the operating system will run the malicious DLL.</p><h3><strong>HANDS-ON</strong></h3><p>We have learned what DLL hijacking is and its attack vectors. Now let's see in practice, from an attacker's point of view, how DLL hijacking is done, by testing it on a virtual machine.</p><p>First we need to pick an application on which we can test the DLL hijacking vulnerability. For this we need to identify files on the system where our read/write permissions are open. There is a resource that will speed this up a great deal: <a href="https://hijacklibs.net/#"><strong>HijackLibs</strong></a>.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*LzJz0BY4EOsh7z_w00KJQw.png" /></figure><p>It is an open-source database that categorizes which programs on the system call which DLLs and the vulnerabilities of those DLLs. From here you can discover potentially vulnerable DLLs, and you can also use the YARA rules the site provides to build rules that automatically detect these weak DLL files and take precautions.</p><p>In this scenario we will use the GitHub tool called <strong>"Siofra"</strong> for DLL hijacking detection and attack.</p><h4><strong>What Is Siofra?</strong></h4><p>Siofra is a tool designed to detect and exploit DLL hijacking vulnerabilities in Windows programs.</p><p>Its key features:</p><ul><li>PE/EXE analysis and dependency extraction</li><li>Simulating which DLLs the application will look for and where, and reporting hijack vulnerabilities.</li><li>Producing a payload-carrying DLL for a targeted DLL name with infect mode.</li><li>Modes for inspecting both on-disk and runtime behavior.</li></ul><h3><strong>1. Vulnerability Analysis</strong></h3><p>Normally, before starting an attack scenario, we perform thorough reconnaissance and information gathering about the target system, both externally and internally. In this scenario, however, our goal will be <strong>to use a vulnerability targeting the Explorer application that is present by default on all Windows systems</strong>. So we will proceed with our steps knowing the vulnerability is present in the Explorer application.</p><p>First, we will run a vulnerability scan with Siofra on the application we suspect has a DLL hijacking vulnerability.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*NafMh6Q5CMVHhdvuRco6lA.png" /><figcaption>Vulnerability scan with Siofra</figcaption></figure><p>This command simulates the DLL loading behavior of the iexplorer.exe file we selected. <strong>It examines which DLLs it searches for and in what order, and checks which directory the target DLL actually resides in.</strong> If the loader searches another directory before the one the DLL lives in, the tool reports that file to us with the "Vulnerable" label.</p><p>In the screenshot, three files appear on which we can perform DLL hijacking. Here we will <strong>try to exploit the iertutil.dll file.</strong> Explorer first looks for iertutil.dll under the <strong>"C:\Program Files\Internet Explorer"</strong> directory. But the original location of iertutil.dll is the <strong>"C:\Windows\System32"</strong> directory.</p><h4><strong>Why Is This a DLL Hijacking Vulnerability?</strong></h4><p>The core problem here is that the application calls the DLL by name only, rather than by full path, and trusts the operating system's default search order.</p><h4><strong>How Is It Exploited?</strong></h4><p>If we create a malicious DLL named iertutil.dll and place it in the location the program looks at first — that is, under "C:\Program Files\Internet Explorer" — then because the loader checks that directory before the original DLL location, it will think it has found the DLL it was looking for and will run our malicious DLL.</p><h3><strong>2. Creating the Malicious DLL</strong></h3><p>To exploit the vulnerability on the target machine, we need to create our own malicious file. In this scenario I will use the msfvenom tool available in Kali Linux to create a DLL payload that opens a reverse shell connection back to the attacker's computer. Since our topic is not how to create payloads, I will only add its screenshot.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/847/1*ZD5hjv-LHgkTFMMLAO9DDQ.png" /><figcaption>Creating a DLL payload with msfvenom</figcaption></figure><h3><strong>3. Placing the Malicious File on the Target Computer</strong></h3><p>If we send our payload to the target computer this way, the payload will run at the end of the process — but because our payload does not perform the work the original iertutil.dll does, Explorer will not run. To solve this, we will use our Siofra tool again;</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*OV4H6LskstLE3FyBMb6e0g.png" /><figcaption>Creating the malicious iertutil</figcaption></figure><p>The command shown in the image takes a copy of our original iertutil.dll file and rebuilds it so that it also runs payload.dll. Finally, we place this new malicious iertutil.dll file under "C:\Program Files\Internet Explorer", the location where the application looks for the DLL first.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/757/1*nsb-Pd4TNRbKC-myBepJGg.png" /></figure><p>As seen in the GIF below, every time Explorer runs, our malicious payload is triggered in the background and provides a reverse shell. Because we wrote over the real DLL instead of running the malicious payload directly, Explorer continued to work normally. With this method it becomes harder for the user to notice our malicious code.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*p893OjcB1T5txM2Rl_tzAQ.gif" /><figcaption>Running the malicious code</figcaption></figure><h3><strong>Prevention and Mitigation Strategies</strong></h3><p>Although it is not always possible to prevent DLL hijacking attacks entirely, the risk can be greatly reduced with the right precautions.</p><ol><li><strong>DLL Verification</strong></li></ol><p>All applications and DLL files should be digitally signed, and the signature value should be verified at runtime. This way the system can detect a DLL file that has been swapped out for the original.</p><p><strong>2. File System Permissions</strong></p><p>User privileges should be restricted in system directories such as C:\Program Files and C:\Windows.</p><p><strong>3. Secure DLL Loading Methods</strong></p><p>DLLs should be loaded using safe APIs such as SetDefaultDllDirectories or LoadLibraryEx. In addition, when DLLs are called, the full file path should be specified rather than leaving it to the default search path.</p><p><strong>4. Software Updates</strong></p><p>The operating system and software should be updated regularly. Files containing known DLL hijacking vulnerabilities should be patched.</p><p><strong>For other mitigation methods for DLL hijacking and more, you can check out the </strong><a href="https://attack.mitre.org/techniques/T1574/001/"><strong>MITRE ATT&amp;CK</strong></a><strong> page.</strong></p><h3><strong>Conclusion</strong></h3><p>Although DLL hijacking is a rarely seen type of attack, when used it is both a simple and effective attack vector. It can be largely prevented with proper engineering and methods. In this post I covered what a DLL is, how attackers exploit these libraries, and defense methods, together with practical examples. I hope it was useful.</p><h3><strong>References</strong></h3><ul><li><a href="https://attack.mitre.org/techniques/T1574/001/">MITRE ATT&amp;CK — T1574.001</a></li><li><a href="https://learn.microsoft.com/en-us/windows/win32/dlls/dynamic-link-library-redirection">Microsoft Docs — DLL Redirection</a></li><li><a href="https://www.crowdstrike.com/en-us/blog/4-ways-adversaries-hijack-dlls/">CrowdStrike — 4 Ways Adversaries Hijack DLLs</a></li><li><a href="https://hijacklibs.net/">HijackLibs — DLL Hijacking Database</a></li></ul>
